## HW 问题收集

在 HW1 和 HW1.1 的过程中，我感觉这两次作业的重点其实不太一样。  
HW1 更偏向于把 `Sudoku` 和 `Game` 这两个领域对象本身写出来，并让它们能通过测试；而 HW1.1 更强调的是，领域对象不能只停留在测试里，而是要真正接入现有的 Svelte 游戏流程，让界面也实际使用它们。所以我遇到的问题，很多都不是单独某个函数怎么写，而是“从 HW1 到 HW1.1 之后，原来的设计为什么不够用了”。

### 已解决

1. 从“测试通过”到“真实界面使用”的差距问题

   1. **上下文**：在 HW1 里，我主要关注的是 `Sudoku` / `Game` 的接口和测试，比如 `guess`、`clone`、`toJSON`、Undo / Redo 等能不能正常工作。但到了 HW1.1，我才发现即使这些测试都通过了，也不代表领域对象已经真的进入了真实游戏流程，因为原来的界面还是在使用旧的 `grid`、`userGrid`、`invalidCells` 等 store。
   2. **解决手段**：我重新梳理了项目里的数据流，重点看了 `src/game.js`、`stores/grid.js`、`stores/game.js` 以及 `Board`、`Keyboard`、`Actions` 这些组件，最后采用了 `gameStore` 作为 adapter/store 的方案。
   3. **解决结果**：现在开始新游戏、输入数字、Hint、Undo / Redo、胜利判断这些真实流程，已经不是直接操作旧数组了，而是统一进入 `gameStore`，再由 `gameStore` 去驱动 `Game` / `Sudoku`。这样我才真正理解了 HW1 和 HW1.1 的差别：HW1 是“对象存在”，HW1.1 是“对象真的被界面使用”。

2. 从“简单对象封装”到“真正的领域对象”的差距问题

   1. **上下文**：HW1 刚开始做的时候，我的 `Sudoku` 更像是一个二维数组的包装器，能 `guess`、能 `clone`、能序列化就已经够用了。但在 HW1.1 中，这种做法明显不够，因为 `Sudoku` 还需要承担更多和规则相关的职责。
   2. **解决手段**：我在原来基础上补充了 `initialGrid`、`fixed`、`solution`，并加入了 `isFixed()`、`getInvalidCells()`、`isWon()` 等能力；同时把 `Game` 的历史管理改成了 `past / future` 的结构，而不是把逻辑散在外面。
   3. **解决结果**：现在的 `Sudoku` 已经不只是“存数组”，而是能表达题面、当前局面、固定格、冲突检测和胜利状态；`Game` 也不再只是测试里临时调用的对象，而是更像一局游戏的统一入口。相比 HW1，我觉得这次最大的提升就是职责边界比以前清楚了。

3. 从“不理解为什么 UI 不刷新”到“理解响应式边界”的问题

   1. **上下文**：HW1 基本是纯 JS 层面的工作，当时不太会碰到“对象变了但界面不刷新”这种问题；但到 HW1.1，我开始真正把领域对象往 Svelte 里接，这时才发现：`Game` / `Sudoku` 内部状态变了，不代表界面一定会跟着更新。
   2. **解决手段**：我去重新理解了 Svelte 3 里的 `store`、`$store`、`derived store` 和直接 mutate 对象之间的区别，并对照自己的 `gameStore` 实现去看。
   3. **解决结果**：我最后理解到，Svelte 不会自动跟踪普通对象内部字段的变化，所以不能指望直接改类实例里的属性就让 UI 刷新。必须把 `gameStore` 作为响应式边界：领域对象变化后，再由 `gameStore` 去更新可订阅状态，组件通过 `$store` 渲染。这个问题在 HW1 里不明显，但在 HW1.1 里是最核心的一个理解点。

### 未解决

1. 现在的接入方式虽然已经完成作业要求，但还有比较强的“过渡性”

   1. **上下文**：为了尽量少改现有组件，我没有一下子把所有旧接口全部删掉，而是保留了 `grid`、`userGrid`、`invalidCells`、`gameWon` 这些原来的 store 名字，只是让它们内部转发到 `gameStore`。
   2. **尝试解决手段**：我采用了 adapter/store 的方式，把真实状态源统一到 `gameStore`，这样既能满足作业要求，也能兼容原来的组件写法。
   3. **当前状态**：这个方案现在是有效的，但我也知道它更像一种“平滑迁移”的做法，而不是最彻底的重构。以后如果继续整理代码，我觉得还可以进一步简化，让组件更直接地使用 `gameStore`，而不是继续依赖这些兼容层。

2. 领域层对现成函数的依赖方式还不够漂亮

   1. **上下文**：为了复用项目里原有的 `printSudoku` 和 `solveSudoku`，我最后是在 `Sudoku.js` 里通过相对路径去导入现成函数文件，而不是完全自己重写。
   2. **尝试解决手段**：中间我尝试过几种导入方式，也遇到过测试环境路径解析失败的问题，最后才调整成现在这个能通过测试的版本。
   3. **当前状态**：现在的做法是可用的，也符合“调用现成函数”的目标，但从模块组织和代码整洁性来看，我觉得还不算特别优雅。如果以后继续改进，我还是希望能找到一种更自然的组织方式。

3. 目前的领域建模已经比 HW1 完整很多，但深度还可以继续提升

   1. **上下文**：相比 HW1，我现在的 `Sudoku` 已经不仅仅是“数组 + guess”，而是加入了题面、固定格、冲突检测、胜利判断等能力。
   2. **尝试解决手段**：我按照 HW1.1 的要求，把更多原本散在外面的规则和状态收回到领域对象内部。
   3. **当前状态**：我认为现在的实现已经能够满足作业要求，但如果从更高质量的领域设计角度看，后面还是可以继续扩展，例如更强的规则表达、更细的合法性检查，或者进一步思考哪些信息应该继续留在 adapter 层，哪些应该再收回领域层。
